# GitHub Trending Tracker — Design

Date: 2026-06-08
Status: Approved for implementation planning

## 1. Goals & Context

A minimal Bun + TypeScript + SQLite tool that periodically scrapes GitHub
trending for daily / weekly / monthly periods and persists each repository's
metadata plus every trending appearance in a single SQLite database. The
database is committed back to the repository by GitHub Actions, giving free
durable storage and full historical replay. Local users query history through
a small CLI.

The reference for shape and scope is
[`/Users/didi/projects/github-rank`](file:///Users/didi/projects/github-rank),
which produces JSON snapshots. This project keeps that flavor (small
script-style entry points, no API token required) but moves storage to SQLite
and adds time-series semantics.

### Non-goals

- No GitHub Search / REST API calls. We only scrape `github.com/trending`.
- No web UI, no API server. CLI-only.
- No language / spoken-language filters in the MVP — the total ranking only.
- No external storage (S3, Turso, …). The committed `.db` file is the
  durable store.

## 2. Architecture Overview

- **Runtime**: Bun (one binary for scripts, CLI, and SQLite driver).
- **Language**: TypeScript (strict), ESM.
- **Database**: SQLite via `bun:sqlite`, accessed through Drizzle ORM
  (`drizzle-orm/bun-sqlite`). Migrations generated by `drizzle-kit`.
- **Scraping**: native `fetch` + `cheerio` against
  `https://github.com/trending?since={daily|weekly|monthly}`.
- **CLI**: a small hand-rolled subcommand dispatcher over `Bun.argv`. No
  third-party CLI library.
- **Scheduling**: GitHub Actions cron, every hour. After each successful run
  the workflow commits `data/trending.db` back to `main`.

### Data flow per run

```
cron tick → bun src/cli.ts fetch
   ├─ scrape trending?since=daily   → ParsedRow[]
   ├─ scrape trending?since=weekly  → ParsedRow[]
   └─ scrape trending?since=monthly → ParsedRow[]
        ↓ single transaction
        for each period:
          UPDATE repo_trending SET is_latest=0 WHERE period=? AND is_latest=1
          for each row:
            UPSERT repo (by full_name)
            UPSERT repo_trending (by period+period_date+full_name)
              → on conflict: refresh rank/stars/today_star,
                             bump update_count, set is_latest=1
   → workflow: git add data/trending.db && commit && push  (only if changed)
```

### Query paths

- `bun src/cli.ts latest --period=daily` → `WHERE period=? AND is_latest=1
  ORDER BY rank` (covered by `idx_trending_latest`).
- `bun src/cli.ts history --repo=owner/name` → `WHERE full_name=? ORDER BY
  captured_at DESC` (covered by `idx_trending_repo`).

### Module boundaries

- `db/*` and `scrape/*` are independent. `commands/*` is the only glue layer.
- `scrape/trending.ts` exports three things: a pure parser
  `parseTrendingHtml(period, html) → ParsedRow[]`, a thin network function
  `fetchTrendingHtml(period, fetchImpl?) → Promise<string>` that wraps
  `fetch` with the retry / timeout policy in §6, and a convenience
  `scrapeTrending(period, fetchImpl?) → Promise<ParsedRow[]>` that
  composes them. The parser never touches the network or the database, so
  fixture-driven tests can call it directly.
- `db/client.ts` exports `openDb(path)`; commands open it on entry, close on
  exit. Tests pass `:memory:`.
- `cli.ts` parses argv and dispatches; it contains no business logic.

## 4. Data Model

### `repo` — one row per repository ever seen

| column            | type                  | notes                              |
|-------------------|-----------------------|------------------------------------|
| `full_name`       | TEXT, PK              | `"karpathy/llm.c"` (global unique) |
| `owner`           | TEXT, NOT NULL        | `"karpathy"`                       |
| `name`            | TEXT, NOT NULL        | `"llm.c"`                          |
| `description`     | TEXT, nullable        | last seen description              |
| `language`        | TEXT, nullable        | last seen primary language         |
| `language_color`  | TEXT, nullable        | hex color string                   |
| `stars`           | INTEGER, default 0    | last seen total stars              |
| `forks`           | INTEGER, default 0    | last seen total forks              |
| `html_url`        | TEXT, NOT NULL        | trending page link                 |
| `first_seen_at`   | INTEGER (unix sec)    | first time it appeared anywhere    |
| `last_seen_at`    | INTEGER (unix sec)    | most recent appearance             |

### `repo_trending` — one row per (period, period_date, repo)

| column              | type                                        | notes                                                          |
|---------------------|---------------------------------------------|----------------------------------------------------------------|
| `id`                | INTEGER PK AUTOINCREMENT                    | surrogate                                                      |
| `full_name`         | TEXT, NOT NULL, FK → repo.full_name         |                                                                |
| `period`            | TEXT, NOT NULL, enum                        | `'daily'` \| `'weekly'` \| `'monthly'`                         |
| `period_date`       | TEXT, NOT NULL                              | `'YYYY-MM-DD'`, UTC; semantics below                           |
| `rank`              | INTEGER, NOT NULL                           | 1..N within batch                                              |
| `today_star`        | INTEGER, NOT NULL                           | parsed numeric from `"… stars today/this week/this month"`     |
| `stars_at_capture`  | INTEGER, NOT NULL                           | total stars at capture moment                                  |
| `forks_at_capture`  | INTEGER, NOT NULL                           | total forks at capture moment                                  |
| `captured_at`       | INTEGER (unix sec), NOT NULL                | most recent UPSERT into this row                               |
| `first_captured_at` | INTEGER (unix sec), NOT NULL                | first time this slot existed                                   |
| `update_count`      | INTEGER, NOT NULL, default 1                | times this slot has been refreshed                             |
| `is_latest`         | INTEGER (boolean), NOT NULL, default 1      | 1 if part of the latest batch for `period`                     |

Indexes:

- `UNIQUE (period, period_date, full_name)` — the slot key.
- `idx_trending_latest (period, is_latest, rank)` — covers `latest`.
- `idx_trending_repo (full_name, period, captured_at)` — covers `history`.

#### `period_date` semantics (UTC)

- `daily` — current UTC date, e.g. `2026-06-08`.
- `weekly` — ISO Monday of the current UTC week, e.g. `2026-06-08`.
- `monthly` — first day of the current UTC month, e.g. `2026-06-01`.

A single shared format keeps queries simple and lets `period_date` be sortable
text without parsing. Rationale for `YYYY-MM-DD` over `YYYY-Www` /
`YYYY-MM`: a single column type for all three periods, no special-casing in
queries.

#### Invariants enforced by writes (§6)

- For any `period`, the rows with `is_latest=1` all share one `period_date`
  (the latest successfully written batch for that period).
- For any `(period, period_date, full_name)` there is at most one row.
- History is never deleted: superseded batches keep their rows with
  `is_latest=0`.

### Capacity estimate

- daily: ~25 distinct rows per day; weekly: ~25 per week; monthly: ~25 per
  month. Within a slot, hourly re-runs only UPSERT (no new rows).
- ≈ 30 new rows/day across periods → ≈ 11k rows/year. With ~100 B per row
  the file stays in the low-MB range per year. Comfortable for git.

## 5. CLI

Single entry: `bun src/cli.ts <subcommand> [flags]`. Three subcommands.

### `fetch` — scrape and write

```
bun src/cli.ts fetch [--db <path>] [--periods daily,weekly,monthly] [--dry-run]
```

- Default scrapes all three periods in order `daily → weekly → monthly`,
  with a 1 s sleep between requests (politeness).
- `--db` defaults to `data/trending.db`. Tests use `:memory:`.
- `--dry-run` parses but does not write; it logs the rows that would be
  written to stderr. Useful for parser debugging.
- Single transaction: any period's parse failure rolls everything back.
- On success, prints a one-line summary per period plus a totals line.

### `latest` — read the latest batch

```
bun src/cli.ts latest [--period daily|weekly|monthly] [--db <path>] [--limit N] [--json]
```

- Defaults: `--period=daily`, `--limit=25`.
- Default output is a wide text table:
  `# | today | stars | repo | lang | description (truncated to 60 chars) |
  captured_at | updates`.
- `--json` prints the raw row array (pipe-friendly for `jq`).
- If no rows have `is_latest=1` for the period: prints
  "no data yet for `<period>` — run `fetch` first" and exits 2.

### `history` — single-repo time series

```
bun src/cli.ts history --repo <owner/name> [--period daily|weekly|monthly|all]
                       [--db <path>] [--limit N] [--json]
```

- `--repo` required. Missing → exit 2 with usage.
- `--period=all` (default) returns rows from all periods.
- Order: `captured_at DESC`. Default `--limit=30`.
- Columns: `captured_at | period | period_date | rank | today_star | stars | updates`.
- No matches → "never seen `<repo>` in trending", exit 2.

### Conventions

- Exit codes: `0` success, `1` runtime/network/parse failure (with stack
  trace), `2` user error or "expected empty result" (missing args, no data).
- `--help` / `-h` on every subcommand and on the root command.
- Times rendered as ISO-8601 UTC.
- Color output suppressed when `process.stdout.isTTY === false`.
- `package.json` scripts wrap the CLI for convenience: `bun run fetch`,
  `bun run latest`, `bun run history`, `bun run db:migrate`,
  `bun run db:generate`, `bun test`, `bun run typecheck`.

## 6. Write Path & Error Handling

### Network layer

- Wrap `fetch` with retry: 3 tries, exponential backoff (1.5 s, 3 s, 6 s).
- 15 s timeout via `AbortController`.
- 5xx and 429 are retryable; other 4xx are fatal.
- Always send a `User-Agent` header (`github-trending-tracker`).

### Parse layer (fail loud)

- `parseTrendingHtml` throws `ParseError` if `.Box-row` is empty, or if any
  required field (`full_name`, `html_url`, `stars_at_capture`,
  `today_star`) is missing on any row.
- Optional fields (`description`, `language`, `language_color`,
  `forks_at_capture`) are allowed to be null/0; the count of missing
  optionals is logged to stderr.
- Parse errors are not retried.

### Transaction & UPSERT

All three periods write inside a single SQLite transaction so the database
either advances fully or not at all.

```ts
db.transaction(() => {
  const now = unixSeconds();
  for (const period of ['daily', 'weekly', 'monthly'] as const) {
    const periodDate = computePeriodDate(period, now);

    // (a) flip the previous latest batch for this period
    db.update(repoTrending).set({ is_latest: false })
      .where(and(eq(repoTrending.period, period),
                 eq(repoTrending.is_latest, true)))
      .run();

    // (b) UPSERT every row in the new batch
    for (const row of parsed[period]) {
      db.insert(repo).values({ /* ... */ }).onConflictDoUpdate({
        target: repo.full_name,
        set: {
          stars: row.stars_at_capture,
          forks: row.forks_at_capture,
          description: row.description,
          language: row.language,
          language_color: row.language_color,
          last_seen_at: now,
        },
      }).run();

      db.insert(repoTrending).values({
        full_name: row.full_name, period, period_date: periodDate,
        rank: row.rank,
        today_star: row.today_star,
        stars_at_capture: row.stars_at_capture,
        forks_at_capture: row.forks_at_capture,
        captured_at: now,
        first_captured_at: now,
        update_count: 1,
        is_latest: true,
      }).onConflictDoUpdate({
        target: [repoTrending.period, repoTrending.period_date,
                 repoTrending.full_name],
        set: {
          rank: row.rank,
          today_star: row.today_star,
          stars_at_capture: row.stars_at_capture,
          forks_at_capture: row.forks_at_capture,
          captured_at: now,
          update_count: sql`${repoTrending.update_count} + 1`,
          is_latest: true,
        },
      }).run();
    }
  }
})();
```

`computePeriodDate`:

```ts
function computePeriodDate(period: Period, nowSec: number): string {
  const d = new Date(nowSec * 1000);
  const utc = (date: Date) => date.toISOString().slice(0, 10);
  if (period === 'daily')   return utc(d);
  if (period === 'monthly') return utc(new Date(Date.UTC(
    d.getUTCFullYear(), d.getUTCMonth(), 1)));
  // weekly: ISO Monday of the current UTC week
  const day = d.getUTCDay();                  // 0=Sun .. 6=Sat
  const offset = day === 0 ? -6 : 1 - day;    // shift to Monday
  return utc(new Date(Date.UTC(
    d.getUTCFullYear(), d.getUTCMonth(), d.getUTCDate() + offset)));
}
```

### Scrape order

All three periods are fully parsed before any writes occur:

```
fetch daily   → parsed.daily
sleep 1s
fetch weekly  → parsed.weekly
sleep 1s
fetch monthly → parsed.monthly
─────────────────────────────────
db.transaction(write all three)
```

### Things deliberately not done (YAGNI)

- No partial-success fallback (e.g. write what we got and skip the rest).
- No request caching.
- No concurrent fetches across periods.
- No "compress old slots" maintenance — the table grows ~11k rows/year.

## 7. Testing Strategy

`bun test` (built-in). Coverage targets are ignored; the focus is the four
failure surfaces below.

### 1. Pure parsers — `tests/parse-today-star.test.ts`

```ts
expect(parseTodayStar('2168 stars today',         'daily')).toBe(2168);
expect(parseTodayStar('1,275 stars this week',    'weekly')).toBe(1275);
expect(parseTodayStar('12,345 stars this month',  'monthly')).toBe(12345);
expect(parseTodayStar('',                          'daily')).toBe(0);
expect(parseTodayStar('  2,168  stars\n today  ', 'daily')).toBe(2168);
```

### 2. HTML scrape — `tests/scrape-trending.test.ts`

Saved fixtures under `tests/fixtures/trending-{daily,weekly,monthly}.html`,
captured during initial implementation. Tests run offline:

- `parseTrendingHtml('daily', fixtureHtml)` returns 20–30 rows whose
  `full_name` matches `/^[\w.-]+\/[\w.-]+$/`, with monotonically increasing
  `rank` starting at 1.
- Empty `<html></html>` throws `ParseError`.
- Optional fields can be null on a synthetic minimal row without error.

When GitHub changes its DOM, the fixture must be refreshed and the test will
make the breakage obvious before the workflow ever pollutes the database.

### 3. DB writes — `tests/db-write.test.ts`

Run against an in-memory SQLite with migrations applied; no real fetch.

- First batch: rows are inserted, all have `is_latest=1`,
  `update_count=1`, `first_captured_at == captured_at`.
- Second batch on a new `period_date`: prior rows for that `period` flip to
  `is_latest=0`; new rows have `is_latest=1`.
- UPSERT on the same `(period, period_date, full_name)`: row count
  unchanged; `update_count` is now 2; `captured_at` is updated;
  `first_captured_at` is unchanged; rank/stars are refreshed.
- Mid-transaction throw: the database is byte-identical to before.

### 4. CLI smoke — `tests/cli.test.ts`

Spawn the real CLI and check stdout / exit codes:

- `latest` on an empty database exits 2.
- `history --repo` missing exits 2.
- `fetch --dry-run` does not modify the file.

No live network in any test.

### `.github/workflows/fetch.yml`

```yaml
name: fetch
on:
  schedule:
    - cron: '17 * * * *'      # every hour at :17, UTC
  workflow_dispatch:
concurrency:
  group: fetch-db
  cancel-in-progress: false
permissions:
  contents: write

jobs:
  fetch:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 1 }
      - uses: oven-sh/setup-bun@v2
      - run: bun install --frozen-lockfile

      # Verify before touching the persistent db.
      - run: bun run typecheck
      - run: bun test

      # Idempotent on subsequent runs.
      - run: bun run db:migrate

      - run: bun src/cli.ts fetch

      - name: Commit & push if changed
        run: |
          git config user.name  "github-actions[bot]"
          git config user.email "41898282+github-actions[bot]@users.noreply.github.com"
          if git diff --quiet -- data/trending.db; then
            echo "::warning::data/trending.db unchanged after fetch"
            exit 0
          fi
          git add data/trending.db
          git commit -m "chore(data): trending snapshot $(date -u +%FT%TZ)"
          git push
```

### `.gitattributes`

```
data/*.db binary
```

### Decisions

- **Hourly cron** (`17 * * * *`) to amortize across timezones and avoid the
  on-the-hour spike seen on shared infrastructure.
- **`concurrency.group=fetch-db`, `cancel-in-progress=false`** — overlapping
  manual triggers and cron ticks queue rather than cancel; we never
  interrupt a half-written transaction.
- **Skip commit when nothing changed** — keeps git log clean. A warning
  surfaces the (unlikely) "completely identical batch" case.
- **No secrets** — only the default `GITHUB_TOKEN` plus `contents: write`.
- **`fetch-depth: 1`** — db is binary and self-contained; no need for full
  history during the workflow.
- **`db:migrate` runs every time** — drizzle migrations are idempotent;
  first run on a new schema version applies it, otherwise no-op.

## 9. Out of Scope (For Later)

- Per-language trending (`?language=…`) and spoken-language filters.
- A query that returns "longest-running on the daily list" or "biggest rank
  gainers this week" — easy to add as new subcommands once the core is in.
- A small static site rendered from the SQLite file (the reference
  project's `template/*.ejs` flow). Deferred until the dataset is mature.
- Periodic compression / archiving of very old `repo_trending` rows. Not
  needed at projected growth.
